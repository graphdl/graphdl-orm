# Order Contracts

This Foundry project deploys and tests the smart contracts generated from `readings/order.md` by the AREST Solidity generator. Every contract here came from compiling FORML 2 readings — nothing is hand-written.

## Prerequisites

Install [Foundry](https://book.getfoundry.sh/getting-started/installation) from a Unix-like shell — macOS Terminal, Linux, WSL, or **Git Bash on Windows**. PowerShell is not supported.

```bash
curl -L https://foundry.paradigm.xyz | bash
foundryup
cd contracts
forge install foundry-rs/forge-std
```

(On older Foundry versions, append `--no-commit` to the install command.)

## What's here

```
contracts/
├── readings/           FORML 2 readings — the source of truth
│   └── order.md        (replace with your own; any number of .md files)
├── src/
│   └── Generated.sol   Solidity generated by `make gen-contracts` (do not hand-edit)
├── test/
│   └── Order.t.sol     Example Foundry tests (hand-written, domain-specific)
├── script/
│   └── Deploy.s.sol    Deploy script generated by `make gen-deploy` (do not hand-edit)
├── foundry.toml        Foundry config (solc 0.8.20, optimizer on)
└── Makefile            Generators + forge targets
```

## Bring your own domain

The shipped `readings/order.md` is an example. Point `make` at any readings directory to regenerate `src/Generated.sol` and `script/Deploy.s.sol` for your own domain:

```bash
# From contracts/
make gen READINGS=../my-app/readings

# Or drop your .md files in contracts/readings and just run
make gen
```

Every entity noun in your readings becomes a deployable contract. Every state machine becomes guarded transition functions. Every constraint becomes a require. The deploy script lists each contract and logs its address after deployment.

## Workflow

```bash
make gen     # Rust engine → readings → Generated.sol + Deploy.s.sol
make build   # forge build
make test    # forge test (6 tests, for the Order domain)
```

When you change readings, regenerate and retest:

```bash
make gen && make test
```

If `make` isn't available, the raw commands are in the Makefile as reference.

## Local deployment (Anvil)

```bash
make anvil           # terminal 1: start the local chain
make deploy-local    # terminal 2: deploy all user contracts
```

Anvil funds 10 accounts with 10k ETH each. The Makefile uses the first for deploys — no wallet setup.

## L2 mainnets

Each target takes `{CHAIN}_RPC_URL` and `PRIVATE_KEY` from the environment. Typical gas cost for six Order operations is well under $1 on any of these.

```bash
# Optimism — public-goods-funded L2, OP Labs governance, low gas
export OPTIMISM_RPC_URL='https://mainnet.optimism.io'
export PRIVATE_KEY='0x...'
```

## What you'll see after deploy

The Order contract has:

- `struct Data { string id; bytes32 status; }` — RMAP-derived storage
- `mapping(string => Data) public records` — keyed by Order ID
- `event OrderWasPlacedByCustomer(string indexed order, string customer)` — fact as event
- `modifier onlyInStatus(string, bytes32)` — state machine guard
- `function create(string memory id)` — UC check + initial status "In Cart"
- `function place(string memory id)` — In Cart → Placed
- `function ship(string memory id)` — Placed → Shipped
- `function archive(string memory id)` — Shipped → Archived

Every transition is guarded by `onlyInStatus`: calling `ship()` on an Order that's still In Cart reverts with "SM: wrong state".

## Round-trip the reading

To verify "the reading is the executable," compare the state machine section of `readings/order.md` to the emitted transition functions in `src/Generated.sol`. They line up one-to-one. Adding a new transition in the reading and running `make gen && make test` will add a new function automatically.

## Known limitations

- The current generator emits contracts for each entity noun separately. Cross-contract references (like `Order was placed by Customer`) are currently only events, not foreign-key style inter-contract calls.
- Deontic constraints do not yet emit require statements; alethic UC and SM guards do.
- Value types (like `Amount`) do not get their own contract; they appear as event parameters and struct fields.

These are improvements on the generator itself; the Foundry project consumes whatever the generator produces, so you'll get them for free when the generator improves.
