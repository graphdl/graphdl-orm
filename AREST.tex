\title{\textbf{Compiling Facts into Applications}}

\author{Samuel Lippert\\Drivly Inc}

\date{}

\begin{document}
\maketitle

\begin{abstract}
Domain knowledge enters software as natural language requirements, is re-encoded as schemas, validation logic, state machines, and API routes, and drifts out of sync in production. The informal layer between specification and execution is where bugs live and where automated agents hallucinate. We show that a single set of FORML\,2 declarations can define a database schema, constraint rules, state machine workflows, and a REST API with hypermedia navigation, producing a complete application from domain facts alone. We formalize this as AREST, a Backus AST system~\cite{backus78} whose definitions are compiled FORML\,2 readings~\cite{halpin08} and whose state is Codd's named set~\cite{codd70} of ORM\,2 elementary facts. REST endpoints are applications of a single system function. Ingesting new readings is self-modification. Five theorems establish that the grammar is unambiguous, that specifications and executables are the same object, that state transfer is complete, that HATEOAS links (both transitions and entity hierarchy) are projections of the population, and that every value in the API is derived from facts. Together they eliminate the informal layer entirely.
\end{abstract}

\section{Introduction}

Software begins as domain knowledge in someone's head. It is re-encoded as schemas, validation logic, state machines, and API routes. When these are distinct artifacts, their equivalence is not preserved by construction.

AREST identifies what was never separate to begin with. A FORML\,2 reading is simultaneously a relation schema, a constraint specification, a resource definition, and an FFP object. The compiler does not translate between representations. It recognizes a single object that already occupies all four roles.

\section{Terminology}

Three terms carry overloaded abbreviations. \emph{AST} follows Backus's usage (Applicative State Transition, 1978), not the compiler term. \emph{ORM} follows Halpin's usage (Object-Role Modeling, 2008), not the persistence pattern. \emph{FP} follows Backus's usage (Function-level Programming), not the broader tradition. \emph{FFP} is Backus's extension of FP with named definitions and a representation function $\rho$ that maps objects to the functions they denote~\cite[Sec.\,13]{backus78}.

\section{Definitions}

Each source formalism solved one layer: Backus the evaluation model, Codd the data model, Halpin the constraint language, and Fielding the transfer protocol. The contribution of AREST is identification, not integration. A FORML\,2 reading is simultaneously an FFP object, a relation schema, a constraint specification, and a resource definition, and the compiler recognizes this identity. The five theorems verify it.

\begin{definition}[AREST]\label{def:arest}
An \emph{Applicative REpresentational State Transfer} system is an AST system~\cite[Sec.\,14]{backus78} whose state $D$ satisfies:
\begin{itemize}[nosep]
\item The $\mathit{FILE}$ cell contains a population $P = \uparrow\!\mathit{FILE}\!:\!D$, a named set of relations~\cite[Sec.\,1.6]{codd70} whose elements are ORM\,2 elementary facts~\cite[Ch.\,2]{halpin08}
\item The $\mathit{DEFS}$ cells contain FFP objects~\cite[Sec.\,13.2]{backus78}. These are readings compiled from FORML\,2 declarations~\cite{halpin08}, and functions registered by the runtime
\end{itemize}
The system function $\mathit{SYSTEM}\!:\!x = \langle o, D'\rangle$~\cite[Sec.\,14.3.1]{backus78} is defined by the definitions in $D$. REST endpoints are applications of $\mathit{SYSTEM}$ with distinct inputs. The representation function $\rho$~\cite[Sec.\,13.3.2]{backus78} maps an object to the function it represents. Given a fact type, $\rho$ looks it up in $\mathit{DEFS}$ and returns the functional form that determines what happens when an operation is applied. The universe of FFP objects $O$ and $\rho$ are inherited from the AST framework.
\end{definition}

A fact type~\cite[Ch.\,2]{halpin08} is a constructor ($\mathit{CONS}$) of roles~\cite[Sec.\,11.2.4]{backus78}. Populate the roles with object instances and you get a fact. The fact is a sequence $\langle\mathit{ft}, o_1, \ldots, o_n\rangle$ where $\mathit{ft}$ is the fact type and $o_1, \ldots, o_n$ are the objects playing its roles. Metacomposition~\cite[Sec.\,13.3.2]{backus78} resolves the fact by looking up its fact type:
\begin{align}
(\rho\langle x_1, \ldots, x_n\rangle)\!:\!y = (\rho\,x_1)\!:\!\langle\langle x_1, \ldots, x_n\rangle, y\rangle \label{eq:metacomp}
\end{align}
$\rho$ resolves the fact type ($x_1$) from $\mathit{DEFS}$ to determine the functional form, and the full fact ($\langle x_1, \ldots, x_n \rangle$) and the operand ($y$) are passed to it. The operand is the operation ($\mathit{render}$, $\mathit{httpFetch}$, $\mathit{validate}$, or any function the runtime has registered). The fact is a function in the $\lambda$-calculus sense: $\lambda f.\, f(o_1, \ldots, o_n)$, accepting an operation and applying it to its own objects.

\noindent Every ORM\,2 constraint compiles to a restriction~(\ref{eq:restrict}) whose predicate belongs to one of two families. The \emph{cardinality} family checks whether a count falls within bounds $[k, m]$: UC sets $[0, 1]$, MC sets $[1, \infty)$, FC sets $[k, m]$. The \emph{membership} family checks whether a tuple exists (or is absent) in a target set. Ring constraints match against the same population with reversed or transitively extended roles. Subset and equality constraints match across distinct fact types. Value constraints match against a declared constant set. Polarity (must exist vs.\ must not exist) is the only parameter distinguishing symmetric (SY, violation when reverse is absent) from asymmetric (AS, violation when reverse is present).

\subsection{Relational Algebra}

Codd's adequate $\theta_1$~\cite[Sec.\,2.2]{codd70} as FFP definitions (all primitives from Table~\ref{tab:ffp}):
\begin{align}
\pi_L(R) &\equiv \alpha\,[s_{i_1}, \ldots, s_{i_k}] : R \label{eq:proj} \\
R \mathbin{*} S &\equiv \mathit{Filter}(\mathit{eq} \circ [s_{\mathit{sh}}]) \circ \mathit{distl} \label{eq:join} \\
R_{L|_M}\!S &\equiv \mathit{Filter}(p) : R \label{eq:restrict} \\
R \cdot S &\equiv \pi_{1s}(R \mathbin{*} S) \label{eq:rcomp} \\
\gamma(R) &\equiv \mathit{Filter}(\mathit{eq} \circ [s_1, s_n]) : R \label{eq:tie}
\end{align}
\noindent where $s_{\mathit{sh}}$ selects the roles shared between $R$ and $S$.

\noindent $P$ is a named set of relations in normal form (ORM\,2 elementary facts are 5NF by construction~\cite[Ch.\,12]{halpin08}). Codd shows that $\theta_1 = \{\text{projection, natural join, restriction, tie}\}$ is adequate for the named set~\cite{codd72}. Equations~(\ref{eq:proj})--(\ref{eq:tie}) express each element of $\theta_1$ as an FFP definition. The combining forms that query the population are the same combining forms that evaluate constraints and derive facts. There is no separate query language. The evaluation algebra is the query algebra.

\subsection{State and Cells}

The state $D$ is a sequence of cells $\langle\mathit{CELL}, n, c\rangle$~\cite[Sec.\,14.3]{backus78}:
\begin{align}
\uparrow\!n\!:\!D &\to c \qquad \text{(fetch)} \label{eq:fetch} \\
\downarrow\!n\!:\!\langle x, D\rangle &\to D' \quad\; \text{(store)} \label{eq:store}
\end{align}
$\mathit{FILE}$ is a cell whose contents is~$P$. RMAP~\cite[Ch.\,17]{halpin08} determines which facts belong to which cell from the schema's uniqueness constraints. The result is a 3NF row: the complete set of facts that depend on one entity's key. Each entity is a cell. The cell stores exactly the facts that RMAP groups under that entity. A cell is what it says it is: a complete object of a type. Cells may contain sub-stores~\cite[Sec.\,14.7]{backus78}. Partitioning is a cell containing a subset of~$P$.

\section{The System Function}

Each input $x$ addresses an entity and carries an operation. The runtime maps the HTTP method to a registered function (\texttt{GET} $\to$ $\mathit{read}$, \texttt{POST} $\to$ $\mathit{create}$, \texttt{PUT} $\to$ $\mathit{update}$, \texttt{DELETE} $\to$ $\mathit{delete}$). The system function routes the input to the addressed entity and passes the operation as the operand:
\begin{align}
\mathit{SYSTEM}\!:\!x &= (\rho\,(\uparrow\!\mathit{entity}(x)\!:\!D))\!:\!\uparrow\!\mathit{op}(x) \label{eq:sys}
\end{align}
The entity is fetched from $D$. $\rho$ resolves it to a functional form determined by its fact type and the definitions in $\mathit{DEFS}$. The operation ($\mathit{read}$, $\mathit{create}$, etc.) is applied as the operand. The entity handles the dispatch, not the system function. In conventional architectures, a router dispatches operations to handlers. Adding a new operation or a new entity type requires changing the router. Here, the entity receives an operation it has never seen before and applies it, because the entity is a $\lambda$-function and the operation is its argument. New operations are registered in $\mathit{DEFS}$ without modifying any entity, and new entity types are added without modifying any operation. This is what makes the system shardable: each entity is self-contained, and two entities on different machines apply the same operation without coordinating.

Self-modification is the same mechanism. Ingesting readings~\cite[Sec.\,14.6]{backus78} is an application of $\mathit{SYSTEM}$ where the addressed entity is $\mathit{DEFS}$ and the operation is $\mathit{compile} \circ \mathit{parse}$. The new FFP objects are stored via $\downarrow\!\mathit{DEFS}$. Subsequent applications of $\mathit{SYSTEM}$ evaluate the new constraints and derivation rules. The system extends its own program.

\subsection{Commands}

$\mu$ denotes evaluation~\cite[Sec.\,14.3.1]{backus78}. Applying $\mathit{SYSTEM}$ to an input produces an output $o$ and a new state $D'$. Each application is an AST transition: the state before, the state after, nothing else changes.

A \textsc{Create} command:
\begin{align}
\mathit{create} &= \mathit{emit} \circ \mathit{validate} \circ \mathit{derive} \circ \mathit{resolve} \label{eq:create}
\end{align}
$\mathit{resolve}$ determines identity from the reference scheme. $\mathit{derive}$ forward-chains derivation rules to the least fixed point. $\mathit{validate}$ applies every constraint as a restriction~(\ref{eq:restrict}) over the complete population, producing violation sets. $\mathit{emit}$ always runs. An alethic violation produces a rejection and the state is unchanged ($D' = D$). A deontic violation produces warnings but the operation succeeds ($D' = D''$). The violation message is the original reading (Corollary~\ref{cor:verbalize}).

A state machine is a left fold over an event stream:
\begin{align}
\mathit{machine}(s_0, E) &= \mathit{foldl}\;\mathit{transition}\;s_0\;E \label{eq:sm} \\
\mathit{transition}(s, e) &= (p \to \bar{s'} ;\; \bar{s})\!:\!\langle s, e \rangle \label{eq:trans}
\end{align}
where $\mathit{foldl}\;f\;z\;\langle\rangle = z$ and $\mathit{foldl}\;f\;z\;\langle e, E'\rangle = \mathit{foldl}\;f\;(f\;z\;e)\;E'$. The transition function is a chain of Backus conditions~\cite[Sec.\,11.2.4]{backus78}: $(p \to f;\; g)$ where $p$ tests whether $\langle s, e\rangle$ matches a declared transition, $f$ returns the target state, and $g$ tries the next condition or returns the current state~$\bar{s}$ as fallback. Events are stored and processed in chronological order. The fold is pure replay. Applying the event stream to a machine produces the current state without side effects, and no entry verbs or callbacks execute. This is restoration of a hibernating machine. Only a new event, not yet in the stream, causes a real transition where the entry verb fires and effects are produced.

Deletion is a transition to a terminal state, not a distinct primitive. The entity receives $\mathit{delete}$ as the operation. It reaches a state with no outgoing transitions and is excluded from query results by restriction~(\ref{eq:restrict}).

\subsection{Platform Binding}

The runtime registers functions into $\mathit{DEFS}$ via $\downarrow\!\mathit{DEFS}$. A browser runtime registers rendering functions, and a server runtime registers $\mathit{httpFetch}$, $\mathit{upsert}$, and $\mathit{notify}$. In practice, metacomposition~(\ref{eq:metacomp}) looks like this:
\begin{align*}
(\rho\,\mathit{fact})\!:\!\mathit{render} &\to \mathit{widget} \\
(\rho\,\mathit{fact})\!:\!\mathit{httpFetch} &\to \mathit{response} \\
(\rho\,\mathit{fact})\!:\!\mathit{upsert} &\to D'
\end{align*}
The fact receives a function and applies it to its own objects. Runtime functions that interact with external state execute during $\mathit{resolve}$~(\ref{eq:create}), populating cells before $\mathit{derive}$ runs, and functions that consume the representation execute after $\mathit{emit}$. This keeps $\mathit{derive}$ pure, monotonic, and terminating over $P$. A single $\mathit{SYSTEM}$ serves browser, server, and storage runtimes by varying $\mathit{DEFS}$, not by varying the logic. Because the binding is via $\rho$, it is live: when a cell's contents change via $\downarrow$, the $\rho$-application re-evaluates and the bound function fires, making event streaming the re-evaluation of $\rho$ over updated $D$ rather than a separate subscription mechanism.

\section{Example}

An order domain is declared in FORML\,2:

\begin{lstlisting}
Order(.OrderId) is an entity type.
Customer(.Name) is an entity type.
Order was placed by Customer.
  Each Order was placed by exactly one
  Customer.
Status 'In Cart' is initial in
  State Machine Definition 'Order'.
Transition 'place' is from
  Status 'In Cart'.
Transition 'place' is to
  Status 'Placed'.
Transition 'ship' is from
  Status 'Placed'.
Transition 'ship' is to
  Status 'Shipped'.
\end{lstlisting}

The constraint reading compiles to a restriction~(\ref{eq:restrict}). $\rho$ yields the following.
\begin{align*}
(\rho\,\mathit{reading})\!:\!P_{\mathit{placedBy}} = \mathit{Filter}(\mathit{count}(s_1) > 1) : P_{\mathit{placedBy}}
\end{align*}
The result is the set of Orders with more than one Customer in $\mathit{placedBy}$. An empty set means the constraint is satisfied.

\texttt{POST /orders \{"customer":"acme"\}} is the application $\mathit{SYSTEM}\!:\!x$ where the addressed entity is $\mathit{orders}$ and the operation is $\mathit{create}$~(\ref{eq:create}). By~(\ref{eq:sys}), $(\rho\,\mathit{orders})\!:\!\mathit{create}$ applies the pipeline:
\begin{align*}
P' = P \cup \{&\mathit{Order}(\texttt{ord\text{-}1}),\;
\mathit{placedBy}(\texttt{ord\text{-}1}, \texttt{acme}),\\
&\mathit{SM}(\texttt{ord\text{-}1}, \texttt{In Cart})\}
\end{align*}

The response includes links derived from $P'$ (Theorem~\ref{thm:hateoas}).
\begin{lstlisting}
{ "id": "ord-1", "customer": "acme",
  "status": "In Cart",
  "_links": {
    "place": {
      "href": "/orders/ord-1/transition",
      "method": "POST" } } }
\end{lstlisting}

Following ``place'' applies~(\ref{eq:sm}). The new representation contains $\{\mathit{ship}\}$, not $\{\mathit{place}\}$, because the status is now Placed and only the outgoing transition from Placed is available. If a second command assigns a different Customer to the same Order, the UC restriction produces a non-empty violation set and the command is rejected. The violation is returned as the original reading (Corollary~\ref{cor:verbalize}).

\section{Execution Semantics}

\subsection{Evaluation}

$\mu$ evaluates by applicative-order $\beta$-reduction~\cite[Sec.\,13.3.1]{backus78}. It repeatedly replaces the innermost application $(x\!:\!y)$ with $(\rho\,x)\!:\!y$, where $\rho$ resolves $x$ to a $\lambda$-term under definitions $D$~\cite[Sec.\,12.9]{backus78}. All evaluation is over finite $P$ with monotonic derivations. Reduction terminates for all requests.

\subsection{Evaluation Strategy}

Three constraints govern execution:
\begin{enumerate}[nosep]
\item \textbf{Reduction order:} applicative (call-by-value). All arguments are reduced before the function is applied.
\item \textbf{Scope:} evaluation is confined to the request-local partition of $D$. RMAP~\cite[Ch.\,17]{halpin08} determines the partition from the schema's uniqueness constraints.
\item \textbf{Derivation:} forward chaining, monotonic, evaluated to the least fixed point per request. Since $\mathit{derive}$ only adds facts over finite $P_{\mathit{local}}$, the fixed point exists (Knaster-Tarski) and is reached in $\leq |P_{\mathit{local,max}}|$ iterations. The derivation chain (which rules fired, in what order, producing which facts) is recorded and available on demand as part of the representation.
\end{enumerate}
The completeness result (Theorem~\ref{thm:complete}) assumes evaluation to the least fixed point under this strategy.

\subsection{Boundary Conditions}

The system operates over facts~$(P)$, computations~$(O$ via $\rho)$, and inputs~$(I)$, the operand $x$ in $\mu(\mathit{SYSTEM}\!:\!x)$~\cite[Sec.\,14.3.1]{backus78}. Time is an input fact in $x$, not hidden state. Effects are directives in the output $o$. Their outcomes are facts in $D'$. Identity follows from the reference scheme~\cite[Ch.\,5]{halpin08}. $\mathit{resolve}$ in~(\ref{eq:create}) applies it. The world assumption per noun~\cite[Ch.\,7]{halpin08} governs incompleteness. Under CWA, a fact not in $P$ is false. Under OWA, it is unknown. Derivation rules for function invocation operate under OWA because the rules capture known reasons to act, not all possible reasons. When the derived course of action is \emph{unknown}, that is itself a reason to escalate to a human or broader-context agent.

\smallskip\noindent\textbf{Remark} (World assumption on populating functions). The theorems are agnostic to the population source. All populating functions are runtime functions registered in $\mathit{DEFS}$. The world assumption determines the interpretation of absence. A function that exhaustively enumerates its domain operates under CWA: if no matching value is found, the fact is definitively absent. A function that cannot exhaustively enumerate its domain operates under OWA: absence of the fact means unknown, not false. Deontic constraints evaluated over OWA-populated facts are sound but not complete. A reported violation is genuine. The absence of a violation is not a guarantee.

\subsection{Isolation}

A state machine~(\ref{eq:sm}) is a left fold over an event stream. Each event is an input $x$. The accumulator is the state $D$. The fold body is $\mu(\mathit{SYSTEM}\!:\!x) = \langle o, D'\rangle$. The application is this fold. The fold is well-defined only if each $\mu$ runs to completion before the next event mutates $D$. Otherwise two concurrent applications of $\mu$ may read the same $D$, and one silently overwrites the other.

Backus states this directly: ``the state cannot change during the computation of system:x''~\cite[Sec.\,14.6(c)]{backus78}. We make the requirement a definition so that theorems can reference it.

\begin{definition}[Cell Isolation]\label{def:isolation}
For each cell $\langle\mathit{CELL}, n, c\rangle$ in $D$, at most one $\mu$ application that writes to $n$ may be in progress at any time. Concurrent $\mu$ applications over disjoint cells are permitted.
\end{definition}

\noindent RMAP~\cite[Ch.\,17]{halpin08} determines which facts belong to which cell from the schema's uniqueness constraints. Each entity is a cell. Its fold is independent of other entities' folds. A single-threaded runtime (e.g.\ a WASM isolate) satisfies Definition~\ref{def:isolation} trivially. A distributed system requires one writer per cell (actor mailbox, partitioned queue, or single-writer lease).

Theorems~\ref{thm:complete}--\ref{thm:derive} hold under Definition~\ref{def:isolation}. Without it, Theorem~\ref{thm:complete} fails silently. Two concurrent $\mathit{create}$ operations on the same cell may both pass $\mathit{validate}$ against stale $P$ and produce a $D'$ that violates a uniqueness constraint. Unlike a grammar violation (Theorem~\ref{thm:grammar}), which is caught at parse time, a cell isolation violation produces inconsistent state without an error.

\subsection{Distributed Evaluation}

Each cell from Sec.\,4.2 stores the 3NF row, including the state machine's current status as one of its facts. The event stream $E$ is demultiplexed into per-cell streams by RMAP~\cite[Ch.\,17]{halpin08}:
\begin{align}
E_n &= \mathit{Filter}(\mathit{eq} \circ [\mathit{RMAP}, \bar{n}]) : E \label{eq:demux}
\end{align}

Each cell folds its stream independently:
\begin{align}
D_n' &= \mathit{foldl}\;\mu_n\;D_n\;E_n \label{eq:cellfold}
\end{align}
where $\mu_n$ is $\mathit{SYSTEM}$ restricted to cell $n$. By Definition~\ref{def:isolation}, no $\mu_n$ reads another cell's in-progress state. Cross-cell queries read committed state:
\begin{align}
P &= \textstyle\bigcup_n \uparrow\!\mathit{FILE}\!:\!D_n \label{eq:pop}
\end{align}

The router distributes events via $\alpha\,\mathit{RMAP}$~(apply-to-all), and each cell reduces its partition via $\mathit{foldl}\;\mu_n$. Adding cells to the cluster scales $\mathit{SYSTEM}$ horizontally without changing its definition.

A browser runtime is a node in the same model. Cells replicated locally fold events with zero latency. The event stream syncs to the server asynchronously. Because the fold is pure and deterministic~(\ref{eq:sm}), two nodes folding the same events reach the same state. Constraints that depend only on the local cell (cardinality, value) evaluate immediately. Constraints that span cells (subset, external uniqueness) evaluate against committed state~(\ref{eq:pop}) after sync. Local cells are untrusted: the server fold is authoritative, and alethic violations on the server reject the event regardless of the local evaluation. Local constraint evaluation is optimistic feedback, not a security boundary.

For two peers sharing a cell, one proposes an event and the other runs $\mathit{validate}$~(\ref{eq:create}) against $P \cup \mathit{resolve}(S, e)$. The event is accepted iff $V$ contains no alethic violation. The constraint set $C_S$ is the consensus predicate (Corollary~\ref{cor:consensus}). For $n > 2$ peers, a shared event log with total ordering is required. Each event references the hash of its predecessor, forming a chain that guarantees integrity and deterministic replay. Peers compare chain tips and adopt the longest chain on fork. The fold~(\ref{eq:cellfold}) over the shared log produces the same state on every peer.

For anonymous peers, three additional properties are required. First, each event carries a cryptographic signature identifying the peer that appended it. Second, reading events ($\mathit{compile} \circ \mathit{parse}$ applications that modify $\mathit{DEFS}$) require authorization. This is a deontic constraint evaluated on every fold: a peer not authorized to modify $\mathit{DEFS}$ can append fact events but not reading events. Third, every peer validates independently on fold, so a malicious event that violates an alethic constraint is detected deterministically by every honest peer. No peer trusts another peer's evaluation. The hash chain provides ordering and tamper detection, the constraints provide validity, the signatures provide identity, and the authorization constraints provide permission. These four properties compose from the existing formalism without a separate security layer.

\section{Properties}

The five theorems form a chain. Each depends on the previous one and eliminates one source of failure in the path from domain knowledge to running software.

\begin{theorem}[Grammar Unambiguity]\label{thm:grammar}
Let $N$ be a declared set of object type names and $F$ a declared set of predicate readings, such that no element of $N$ contains a formal item of the grammar as a substring. Then each FORML\,2 sentence over $(N, F)$ has exactly one parse.
\end{theorem}

\begin{proof}
FORML\,2 sentences fall into three structural families, each determined by its prefix:

\smallskip\noindent
\textit{(a) Quantified constraints.} Structure:
$[\mathit{modal}] \;\; \mathit{quant}_1 \;\; n_1 \;\; r \;\; \mathit{quant}_2 \;\; n_2 \;\; [\mathit{qual}]$

\noindent where $\mathit{modal} \in \{$``It is obligatory that'', ``It is forbidden that'', ``It is permitted that'', ``It is impossible that'', ``It is possible that'', $\varepsilon\}$; $\mathit{quant}_1 \in \{$``Each'', ``For each'', ``the same'', ``No'', $\varepsilon\}$; $\mathit{quant}_2 \in \{$``at most one'', ``some'', ``exactly one'', ``at least $k$ and at most $m$'', ``more than one'', ``no'', $\varepsilon\}$. Noun positions $n_i \in N$ are resolved by longest-first matching. The predicate $r \in F$ is the maximal token sequence between noun positions. Each structural position admits exactly one parse because (i)~modal prefixes are pairwise non-overlapping, (ii)~quantifier phrases are pairwise distinct, (iii)~no $n \in N$ contains a formal item as a substring (by hypothesis), and (iv)~longest-first matching is deterministic. The constraint kind is determined by $(\mathit{quant}_1, \mathit{quant}_2)$: (Each, at most one)$\,\to\,$UC; (Each, some)$\,\to\,$MC; (Each, exactly one)$\,\to\,$UC$+$MC; (Each, at least $k$\ldots)$\,\to\,$FC; (the same, more than one)$\,\to\,$UC$^{-}$; (No, $\varepsilon$)$\,\to\,$XC. This family also generalizes to $n$-ary predicates ($n_1 \ldots n_k$) by the same longest-first procedure.

\smallskip\noindent
\textit{(b) Conditional constraints.} Structure:
$\text{``If''} \;\; \mathit{antecedent} \;\; \text{``then''} \;\; \mathit{consequent}$

\noindent The ``If\ldots then'' frame is unambiguous (unique split on ``then''). Ring constraints (IR, AS, SY, AT, IT, TR, AC, RF) are identified by all noun bases being the same type. The kind is determined by polarity (``it is impossible that'' in the consequent) and arity (``and'' in the antecedent). Subset constraints (SS) are identified by existential ``some'' in the antecedent and anaphoric ``that'' in the consequent with mixed noun types. Derivation rules occupy the remainder.

\smallskip\noindent
\textit{(c) Multi-clause constraints.} Keyword-delimited: ``if and only if'' (EQ), ``exactly one of the following holds'' (XO), ``at most one of the following holds'' (XC), ``at least one of the following holds'' (OR). The keyword is unambiguous. Clauses are separated by ``;'' or ``or''.

\smallskip\noindent
The three families are distinguished by their sentence prefix (quantifier word, ``If'', or keyword), which is determined at position~(1). Within each family, the parse is deterministic by the arguments above. The reference implementation~\cite{norma} constructively demonstrates unambiguous round-trip parsing for all 17 constraint kinds.
\end{proof}

\noindent The hypothesis on $N$ is enforced by the compiler, which rejects any declared name containing a formal item of the grammar as a substring.

\begin{theorem}[Specification Equivalence]\label{thm:spec}
Let $\mathit{parse}: R \to \Phi$ and $\mathit{compile}: \Phi \to O$ be injective, with $R$ the set of FORML\,2 readings, $\Phi$ the set of FOL formulas, and $O$ the set of FFP objects. Then:
\begin{align}
\mathit{parse}^{-1} \circ\, \mathit{compile}^{-1} \circ\, \mathit{compile} \circ \mathit{parse} &= \mathit{id}_R \label{eq:spec}
\end{align}
and $\rho(\mathit{compile}(\mathit{parse}(r)))$ is the executable for every $r \in R$.
\end{theorem}

\begin{proof}
$\mathit{parse}$ is injective by Theorem~\ref{thm:grammar}: each sentence has one parse, so distinct sentences yield distinct formulas. $\mathit{compile}$ is injective: each constraint kind compiles to a restriction~(\ref{eq:restrict}) with a predicate from one of two families: cardinality predicates parameterized by bounds $[k, m]$, and membership predicates parameterized by target set and polarity. Distinct kinds produce distinct parameter tuples, and thus distinct FFP objects. Both $\mathit{parse}$ and $\mathit{compile}$ have left inverses ($\mathit{compile}^{-1}$ is Halpin and Curland's verbalization algorithm~\cite{halpin06}); composing: $\mathit{parse}^{-1} \circ \mathit{compile}^{-1} \circ \mathit{compile} \circ \mathit{parse} = \mathit{id}_R$. Executability: $\rho$ maps the compiled object to a function by~\cite[Sec.\,13.4]{backus78}.
\end{proof}

\noindent\textbf{Remark} (Surjectivity). $\mathit{compile}$ is not surjective. There exist FFP objects in $O$ that no FORML\,2 reading produces. This is intentional: runtime-registered functions (Sec.\,5.3) occupy the complement. FORML\,2 readings cover the domain layer. Registered functions cover the platform layer. Together they span the objects in $\mathit{DEFS}$.

\begin{theorem}[Completeness of State Transfer]\label{thm:complete}
Let $F_S : 2^P \to 2^P$ be the monotonic operator that applies all derivation rules in schema $S$ to a population, and let $C_S$ be the set of constraint objects in $S$. Under Definition~\ref{def:isolation}, $\mathit{create}$~(\ref{eq:create}) applied to input $I$ and state $D$ produces $\langle o, D'\rangle$ such that:
\begin{enumerate}[nosep]
\item $P' = P \cup \mathit{resolve}(S, I)$ contains all entity facts entailed by $(S, I)$;
\item $P'' = \mathit{lfp}(F_S, P')$, the least fixed point of $F_S$ above $P'$;
\item $V = \bigcup\{(\rho\,c)\!:\!P'' \mid c \in C_S\}$ collects all violations (each constraint produces a possibly empty set via restriction);
\item $o$ is the REST representation of $(P'', V, \mathit{links}(s))$ and $D' = D[\mathit{FILE} \mapsto P'']$ if $V$ contains no alethic violation, else $D' = D$.
\end{enumerate}
\end{theorem}

\begin{proof}
$P$ is finite. $F_S$ is monotonic (derivation rules only add facts). By Knaster-Tarski, $\mathit{lfp}(F_S, P')$ exists and is unique. Iterative application of $F_S$ reaches it in at most $|P''_{\max}| - |P'|$ steps. $\mathit{validate}$ applies every $c \in C_S$ as a restriction over the fixed point; each produces a (possibly empty) set of violations. $\mathit{emit}$ constructs $o$ from $P''$, $V$, and $\mathit{links}(s)$ (Theorem~\ref{thm:hateoas}). The state does not change during evaluation~\cite[Sec.\,14.6(c)]{backus78}; Definition~\ref{def:isolation} ensures no concurrent write to the same cell.
\end{proof}

\begin{theorem}[HATEOAS as Projection]\label{thm:hateoas}\label{thm:nav}
All links in the representation are $\theta_1$ operations on $P$ and $S$.

\smallskip\noindent
\textit{(a) Transition links.} Let $T = \{(s_i, s_j, e_k)\} \subseteq P$ be transition facts and $s$ the current status. Then:
\begin{align}
\mathit{links}(s) &= \pi_{\mathit{event}}\!\left(\mathit{Filter}(p) : T\right) \label{eq:hateoas}
\end{align}
where $p(t) = (s_{\mathit{from}}(t) = s) \lor \mathit{anc}(s_{\mathit{from}}(t), s)$ and $\mathit{anc}(s', s)$ holds when $s$ is reachable from $s'$ through the transition graph.

\smallskip\noindent
\textit{(b) Navigation links.} Let $F_B$ be the binary fact types between entity nouns. For each $f \in F_B$, let $r_c$ denote the role constrained by a UC and $\bar{r}_c$ the other role. A UC on $r_c$ establishes a functional dependency from $r_c$ to $\bar{r}_c$, making the $r_c$ noun the child. The navigation links for an entity $e$ of noun $n$ are:
\begin{align}
\mathit{children}(n) &= \{\mathit{noun}(r_c) : f \in F_B,\; \mathit{uc}(r_c),\; \mathit{noun}(\bar{r}_c) = n\} \label{eq:children} \\
\mathit{parent}(n) &= \{\mathit{noun}(\bar{r}_c) : f \in F_B,\; \mathit{uc}(r_c),\; \mathit{noun}(r_c) = n\} \label{eq:parent} \\
\mathit{nav}(e, n) &= \mathit{children}(n) \;\cup\; \mathit{parent}(n) \label{eq:nav}
\end{align}

\smallskip\noindent
The full link set combines both:
\begin{align}
\mathit{links}_{\mathit{full}}(e, n, s) &= \mathit{nav}(e, n) \;\cup\; \mathit{links}(s) \label{eq:linksfull}
\end{align}
\end{theorem}

\begin{proof}
\textit{(a)} $T \subseteq P$. $\mathit{Filter}$ is restriction~(\ref{eq:restrict}); $\pi$ is projection~(\ref{eq:proj}). Both are $\theta_1$ operations.
\textit{(b)} $F_B \subseteq S$. Each set comprehension is a projection over a restriction of $F_B$, both $\theta_1$ operations. The UC on $r_c$ makes $r_c$ functionally dependent on $\bar{r}_c$ (RMAP step~2). The entity hierarchy is determined by this projection without external configuration. Both components of $\mathit{links}_{\mathit{full}}$ are $\theta_1$ operations on $S$ and $P$.
\end{proof}

\begin{corollary}[Deletion as Terminal State]\label{cor:delete}
Let $s_{\mathit{del}}$ be a state with no outgoing transitions in $T$, i.e.\ $\neg\exists\,t \in T : s_{\mathit{from}}(t) = s_{\mathit{del}}$. Then $\mathit{links}(s_{\mathit{del}}) = \emptyset$ by Theorem~\ref{thm:hateoas}, and the entity is excluded from query results by $\mathit{Filter}(p_{\mathit{live}}) : P$ where $p_{\mathit{live}}(e) = \exists\,t \in T : s_{\mathit{from}}(t) = \mathit{status}(e)$.
\end{corollary}

\begin{theorem}[Derivability]\label{thm:derive}
Define the \emph{representation} of an entity $e$ in population $P$ under schema $S$ as:
\begin{align*}
\mathit{repr}(e, P, S) = \;&\{(\rho\,s_i)\!:\!f \mid f \in P,\, f \text{ involves } e\} \\
\cup\; &\{(\rho\,r)\!:\!P \mid r \in S_{\mathit{derivations}}\} \\
\cup\; &\{(\rho\,c)\!:\!P \mid c \in S_{\mathit{constraints}}\} \\
\cup\; &\mathit{links}_{\mathit{full}}(e, n, \mathit{status}(e, P))
\end{align*}
where $\mathit{status}(e, P)$ is the current status of $e$, a fact in $P$ (absorbed into the entity's cell by RMAP), and $\mathit{links}_{\mathit{full}}$ combines navigation links (Theorem~\ref{thm:nav}) with transition links (Theorem~\ref{thm:hateoas}). Then for every $v \in \mathit{repr}(e, P, S)$, there exists $f \in O$ such that $v = (\rho\,f)\!:\!P$.
\end{theorem}

\begin{proof}
Each component of $\mathit{repr}$ is a $\rho$-application by construction. Selectors extract attributes. Derivation rules are FFP objects whose $\rho$-image computes derived facts. Constraints are restrictions~(\ref{eq:restrict}) whose $\rho$-image produces a set of violations (empty if satisfied). Links are $\theta_1$ operations (Theorem~\ref{thm:hateoas}). No value in $\mathit{repr}$ is produced by a mechanism outside $\rho$.
\end{proof}

\begin{corollary}[Violation Verbalization]\label{cor:verbalize}
Let $c \in C_S$ be a constraint object compiled from reading $r \in R$. If $(\rho\,c)\!:\!P'' \neq \emptyset$, the violation output is $\mathit{compile}^{-1}(c)$, recovering $r$ by Theorem~\ref{thm:spec}. The error message is the original reading.
\end{corollary}

\noindent Each $\theta_1$ primitive has a FORML\,2 verbalization. Projection verbalizes as role selection, join as qualified composition, restriction as constraint predicate, tie as self-reference. Halpin and Curland~\cite{halpin06} give the automated verbalization algorithm for ORM\,2 constraints. Composed $\theta_1$ operations verbalize via FORML\,2 qualified readings (``who\ldots'', ``that\ldots''), which parse unambiguously by Theorem~\ref{thm:grammar}.

\begin{corollary}[Closure Under Self-Modification]\label{cor:closure}
Let $D' = D$ after applying $\mathit{SYSTEM}$ with operation $\mathit{compile} \circ \mathit{parse}$ to a set of FORML\,2 readings $R'$. Then Theorems~\ref{thm:grammar}--\ref{thm:derive} hold for all subsequent applications of $\mathit{SYSTEM}$ over~$D'$.
\end{corollary}

\begin{proof}
Each new reading $r' \in R'$ parses unambiguously (Theorem~\ref{thm:grammar} depends on the grammar, not on $D$). The compiled objects are faithful to the readings (Theorem~\ref{thm:spec} depends on the injectivity of $\mathit{parse}$ and $\mathit{compile}$, which are stateless). The new constraints and derivation rules enter $\mathit{DEFS}$ via $\downarrow\!\mathit{DEFS}$, joining the existing definitions. Theorem~\ref{thm:complete} evaluates all $c \in C_S$, including the new ones. Theorems~\ref{thm:hateoas} and~\ref{thm:derive} operate over $P$ and $S$, both of which now include the new facts and schemas. No theorem's proof depends on $D$ being fixed at initialization.
\end{proof}

\begin{corollary}[Constraint Consensus]\label{cor:consensus}
For two peers $A$ and $B$ sharing population $P$, if $A$ proposes event $e$ and $B$ applies $\mathit{validate}$~(\ref{eq:create}) to $P \cup \mathit{resolve}(S, e)$, the event is accepted iff $V$ contains no alethic violation. The constraint set $C_S$ is the consensus predicate. No external consensus protocol is required.
\end{corollary}

\noindent For $n = 2$, the constraints are sufficient for consensus. For $n > 2$, a total ordering over the shared event log is additionally required so that all peers fold~(\ref{eq:cellfold}) in the same order.

\medskip
\noindent Common operational concerns map directly to the formalism: versioning is the event stream ($E$), migration is ingestion of new readings (Corollary~\ref{cor:closure}), backward compatibility is preserved by construction (no theorem depends on $D$ at initialization), and auditability is the derivation trace (Sec.\,7.2).

The algebraic laws of FP~\cite[Sec.\,12.2]{backus78} are equations between functional forms. Since FFP objects represent the same forms via $\rho$~\cite[Sec.\,13.3.2]{backus78}, an FFP object can be transformed to an equivalent object by applying these laws. Ahead-of-time compilation from objects to optimized evaluation trees is one such transformation.

\section{Middleware Elimination}

Traditional middleware (authentication, validation, rate limiting, transformation) is either a restriction over $P$, a COMP derivation, or a fact in $P$. Authorization, for instance, is a derivation. ``User accesses Domain iff User belongs to Organization that manages Domain'' compiles to $\langle\mathit{COMP}, f, g\rangle$ whose $\rho$-image computes the access predicate over $P$. External service calls, database persistence, and UI rendering are not middleware either. They are runtime functions registered into $\mathit{DEFS}$ and applied via $\rho$ (Sec.\,5.3). Theorem~\ref{thm:derive} leaves no room for a computation layer outside $\rho$. Every value in the representation is a $\rho$-application.

\section{Conclusion}

AREST is a Backus AST system~\cite{backus78} whose definitions are compiled FORML\,2 readings~\cite{halpin08} and whose state is Codd's named set~\cite{codd70} of ORM\,2 elementary facts. REST endpoints~\cite{fielding00} are applications of the system function. Ingesting readings is self-modification. Transitions produce hypermedia. The five theorems form a chain: the grammar is unambiguous (Theorem~\ref{thm:grammar}), so specifications compile faithfully to executables (Theorem~\ref{thm:spec}), so state transfer is complete (Theorem~\ref{thm:complete}), so all links (transitions and entity hierarchy) are projections of the population (Theorem~\ref{thm:hateoas}), so every value in the representation is a $\rho$-application (Theorem~\ref{thm:derive}). Self-modification preserves all five (Corollary~\ref{cor:closure}). The system grows by ingesting facts and never loses its guarantees.

The five properties compose into a guarantee for automated agents, addressing two distinct sources of LLM hallucination. First, the specification-execution gap. An LLM writing FORML\,2 readings produces unambiguous specifications (Theorem~\ref{thm:grammar}) that are the executables (Theorem~\ref{thm:spec}). An LLM consuming the API reads values that are $\rho$-applications over $P$ (Theorem~\ref{thm:derive}) and discovers every valid action via HATEOAS links (Theorem~\ref{thm:hateoas}). There are no undocumented endpoints, no hidden middleware, no informal contracts to hallucinate through. Second, the reasoning chain gap. LLMs do not reliably chain logical inferences, and errors compound over steps. The forward chainer in $\mathit{derive}$~(\ref{eq:create}) evaluates derivation rules to the least fixed point (Theorem~\ref{thm:complete}), and the derived facts appear in the representation (Theorem~\ref{thm:derive}). The LLM reads \emph{conclusions}, not premises. The logical chain that produced them was evaluated deterministically by the engine, outside the LLM's context. The LLM does not reason through the chain; it consumes the result. When the chain itself is needed (for debugging, auditing, or explanation), the derivation trace is available on demand as a sequence of rule applications over $P$, each a $\rho$-application (Theorem~\ref{thm:derive}).

The gap between domain knowledge and running software is the gap where requirements drift, implementations diverge, and production breaks. AREST eliminates the gap by identifying what was never separate to begin with.
